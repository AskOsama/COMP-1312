1. How Can Poor Table Design Affect a Database?

Key Question: What happens when a database table is not designed properly? Poor table design can lead to duplicate data, inconsistent records, and difficulty retrieving information.

Good table design is the foundation of every reliable database. Without proper design, data becomes disorganized and unreliable, making the database difficult to use and maintain.

Think About It

Imagine a university stores student data without unique student IDs. Consider the following problems:

  • Duplicate entries: Two students named "Ahmed Ali" cannot be distinguished
  • Data confusion: Updating one student's grade may accidentally change another's
  • Lost records: Without a unique identifier, records can be mixed up or lost
  • Inefficient searches: Finding a specific student becomes slow and unreliable

3. Introduction to Tables

What is a Table?

A table is a structured collection of related data organized in rows and columns. Tables are the primary objects used to store data in a relational database.

Properties of a Table

Columns (Fields)

Each column represents a specific attribute or characteristic of the data (e.g., Name, Age, ID)

Rows (Records)

Each row represents a single, complete entry containing values for all the fields

Table Name

Every table has a unique, descriptive name that identifies the type of data it stores

Data Values

Each cell contains a single data value at the intersection of a row and a column

Example: Students Table

StudentID Name Major GPA
1001 Ahmed Ali Computer Science 3.8
1002 Sara Mohammed Information Systems 3.5
1003 Khalid Omar Computer Science 3.9

This table has 4 fields (columns) and 3 records (rows).

4. What is a Field?

Field Definition

A field is a single column in a table that represents a specific attribute of the data. Each field has a name, data type, size, and description.

Properties of a Field

Field Name

A unique, descriptive label that identifies the field (e.g., StudentID, FirstName)

Data Type

Defines what kind of data the field can hold (text, number, date, etc.)

Field Size

The maximum amount of data the field can store (e.g., 50 characters for a name)

Description

An optional explanation of the field's purpose and what data it should contain

Common Data Types

📝

Text

Stores letters, numbers, and symbols as text (e.g., names, addresses)

🔢

Number

Stores numeric values for calculations (e.g., age, quantity, GPA)

📅

Date/Time

Stores dates and times (e.g., birth date, enrollment date)

✅

Yes/No

Stores boolean values — true/false, yes/no (e.g., IsActive, IsGraduated)

5. Field Naming Rules

Choosing proper field names is essential for creating a clear and maintainable database. Follow these rules when naming fields:

Be Descriptive

Use names that clearly describe the data stored in the field (e.g., "StudentName" not "SN")

Be Consistent

Follow the same naming convention throughout the database (e.g., always use CamelCase)

No Spaces

Use underscores or CamelCase instead of spaces (e.g., "FirstName" or "first_name")

No Reserved Words

Avoid using database keywords like "Table", "Select", "Date" as field names

Good vs Poor Field Names

Good Names Poor Names Reason
StudentID ID Too vague — which ID?
FirstName FN Abbreviation is unclear
EnrollmentDate Date "Date" is a reserved word
PhoneNumber Phone Number Contains a space

6. What is a Record?

Record Definition

A record is a single row in a table that contains a complete set of related data values. Each record represents one instance of the entity described by the table.

Example: One Student Record

StudentID Name Major GPA EnrollmentDate
1001 Ahmed Ali Computer Science 3.8 2024-09-01

This single row is one record. It contains all the information about one specific student. Each value in the row corresponds to a field (column) in the table.

Remember

A record must have a value for each field in the table. Together, all records in a table form the complete dataset for that entity (e.g., all students in the university).

7. Importance of Table Design

The quality of a database depends heavily on how well its tables are designed. Good table design ensures data is stored efficiently and accurately.

Good Design

Data is organized, easy to search, free from unnecessary duplication, and maintains integrity

Poor Design

Data is duplicated, difficult to update, prone to errors, and wastes storage space

Warning: Consequences of Poor Table Design

Poor table design can lead to:

  • Data redundancy — the same data stored in multiple places
  • Update anomalies — changing data in one place but not another
  • Insertion anomalies — inability to add data without other unrelated data
  • Deletion anomalies — losing important data when deleting a record
  • Inconsistent data — conflicting information across the database

9. Primary Key

Primary Key Definition

A primary key is a field (or combination of fields) in a table that uniquely identifies each record. No two records can have the same primary key value.

Properties of a Primary Key

Unique

Each value must be different — no two records share the same primary key

Not Null

The primary key field cannot be left empty — every record must have a value

Unchanging

Once assigned, the primary key value should not change over time

Single Value

Each primary key field holds exactly one value per record (atomic)

Examples of Primary Keys

Table Primary Key Why It Works
Students StudentID Each student has a unique ID number
Employees EmployeeID Each employee has a unique ID number
Products ProductCode Each product has a unique code
Courses CourseNumber Each course has a unique number

Tip

Names, phone numbers, and email addresses should not be used as primary keys because they can change or be shared by multiple people. Use a system-generated ID instead.

10. Importance of Primary Key

Why are primary keys essential? Primary keys are the backbone of a well-designed database. They ensure that every record can be uniquely identified and accessed.

Ensures Uniqueness

Prevents duplicate records from being entered into the table

Maintains Data Integrity

Guarantees that each record is distinct and can be reliably referenced

Builds Relationships

Enables linking related tables together through foreign keys

Efficient Data Access

Speeds up searching and retrieving specific records from the database

11. Table Operations

Once a table is created, you need to perform operations to manage the data stored in it. The three fundamental operations are:

➕

Insert

Add a new record to the table (e.g., register a new student)

✏️

Update

Modify existing data in a record (e.g., change a student's GPA)

🗑️

Delete

Remove a record from the table (e.g., remove a withdrawn student)

Examples of Table Operations

  • Insert: A new student enrolls in the university — add their record to the Students table
  • Update: A student changes their major from "Mathematics" to "Computer Science" — update the Major field
  • Delete: A student withdraws from the university — remove their record from the table

Important: Be Careful with Delete

Deleting a record is permanent. Always verify that you are removing the correct record before performing a delete operation. Consider keeping backup copies of data before making deletions.

13. Table Handling

Proper table handling ensures that data remains accurate, secure, and accessible. Follow these best practices:

Validate Data

Check that data entered into the table meets the required format and rules before saving

Maintain Integrity

Ensure all records are complete, consistent, and free from errors or conflicts

Backup Regularly

Create regular backup copies of tables to prevent data loss due to errors or failures

Control Access

Restrict who can view, add, modify, or delete data to maintain security

Warning: Common Table Handling Mistakes

Avoid these common mistakes when working with tables:

  • Entering data without validation — may introduce errors and inconsistencies
  • Not using primary keys — makes it impossible to uniquely identify records
  • Skipping backups — risks permanent data loss
  • Giving unrestricted access — compromises data security and integrity
  • Ignoring data types — storing text in number fields causes errors

Summary and Key Takeaways

What We've Learned

  1. Tables: The primary structure for storing data in a database, organized in rows (records) and columns (fields).
  2. Fields: Columns that represent specific attributes of the data, each with a name, type, size, and description.
  3. Records: Rows that contain a complete set of related data values for one entity instance.
  4. Field Naming: Names should be descriptive, consistent, without spaces, and avoid reserved words.
  5. Table Design: Good design prevents redundancy, anomalies, and maintains data integrity.
  6. Primary Key: A unique identifier for each record — must be unique, not null, unchanging, and atomic.
  7. Primary Key Importance: Ensures uniqueness, maintains integrity, enables relationships, and speeds up access.
  8. Table Operations: Insert (add), Update (modify), and Delete (remove) are the three fundamental operations.
  9. Table Handling: Validate data, maintain integrity, backup regularly, and control access.
  10. Design Matters: A well-designed table is the foundation of a reliable and efficient database system.
