1. Introduction: Why Can't One Table Do Everything?

Key Question: What happens when you try to store all your data in a single table? Imagine storing students, courses, and instructors all in one table.

This design would cause repeated data, confusion, and errors. Relationships between tables help us organize data correctly and maintain accuracy in database systems.

Think About It

If a university stores student names, course names, and instructor names all in one table:

  • Data repetition: The instructor's name is repeated for every student in their course
  • Update problems: Changing an instructor's name requires updating many rows
  • Confusion: It becomes unclear which data belongs to which entity
  • Errors: Inconsistencies arise when data is duplicated across rows

3. Introduction to Relationships

What is a Relationship?

In databases, data is often divided into multiple tables to avoid redundancy and improve organization. A relationship is a logical connection between two tables that allows data to be stored efficiently and retrieved correctly.

Why Relationships Are Needed

Relationships help to:

Avoid Data Repetition

Prevent repeating the same data in multiple tables, saving storage and reducing errors

Maintain Data Consistency

Ensure that changes in one place are reflected correctly across the database

Connect Related Information

Link data from different tables so it can be combined when needed

Support Accurate Data Retrieval

Enable correct and efficient querying of data across multiple tables

Warning

Without relationships, databases become disorganized and unreliable. Data may be duplicated, inconsistent, or impossible to query correctly.

4. Concept of Related Tables

When Are Two Tables Related?

Two tables are related when they share a common field, or when one table depends on another for information.

Example: Students and Courses

Consider a university database with two tables:

Students Table
StudentID Name Major
1001 Ahmed Ali Computer Science
1002 Sara Mohammed Information Systems
Courses Table
CourseID CourseName Credits
CS101 Database Fundamentals 3
CS102 Programming 1 4

A student can register for courses, which creates a relationship between the two tables.

5. Keys and Relationships

Key Concept: Relationships between tables are created using keys. One table contains a Primary Key, and another table contains a Foreign Key that references it. Keys allow tables to be linked correctly.

🔑

Primary Key (PK)

A field that uniquely identifies each record in its own table

🔗

Foreign Key (FK)

A field in one table that refers to the primary key in another table

6. Primary Key in Relationships

The Primary Key plays a central role in establishing relationships between tables:

Uniquely Identifies Each Record

Every record in the table has a distinct primary key value that sets it apart from all other records

Ensures Each Record Can Be Distinguished

No two records can share the same primary key value, preventing ambiguity

Is Referenced by Other Tables

Other tables use this key as a foreign key to create links and establish relationships

Essential for Building Relationships

Without primary keys, it would be impossible to reliably connect related tables

Example

In the Departments table, DepartmentID is the primary key. Each department has a unique ID that other tables can reference.

7. Foreign Key in Relationships

Foreign Key Definition

A Foreign Key is a field in one table that refers to the primary key in another table. It creates a link between the two tables, allowing them to share information.

Creates a Link Between Tables

The foreign key establishes a connection from one table (child) to another (parent)

Is a Field in One Table

The foreign key exists as a regular field in the child table

Refers to the Primary Key in Another Table

Its values must match existing primary key values in the referenced (parent) table

Example: Students and Departments

Departments Table (Parent)
DepartmentID (PK) DepartmentName
D01 Computer Science
D02 Information Systems
Students Table (Child)
StudentID (PK) Name DepartmentID (FK)
1001 Ahmed Ali D01
1002 Sara Mohammed D02

The DepartmentID in the Students table is a foreign key that references the DepartmentID primary key in the Departments table.

Remember

Foreign keys allow tables to share information. They ensure that every reference in the child table points to a valid record in the parent table.

8. Types of Relationships

There are three main types of relationships in databases. Each type is used based on the real-world situation being modeled:

1:1

One-to-One (1:1)

Each record in one table is linked to only one record in another table

1:M

One-to-Many (1:M)

One record in a table can be linked to many records in another table

M:M

Many-to-Many (M:M)

Many records in one table can relate to many records in another table

One-to-One Relationship (1:1)

Definition

Each record in one table is linked to only one record in another table

Usage

Used rarely in databases. Often applied for security or optional data separation

Example: Person and Passport

Each person has exactly one passport, and each passport belongs to exactly one person. This is a One-to-One (1:1) relationship.

9. One-to-Many Relationship (1:M)

Most Common: The One-to-Many relationship is the most common type of relationship in database design.

One-to-Many Definition

In a One-to-Many (1:M) relationship, one record in the parent table can be associated with many records in the child table, but each record in the child table is associated with only one record in the parent table.

Example: Departments and Students

Relationship Description
One department has many students
Each student belongs to one department

The foreign key (DepartmentID) is placed in the Students table (the "many" side).

10. Many-to-Many Relationship (M:M)

Many-to-Many Definition

In a Many-to-Many (M:M) relationship, many records in one table can relate to many records in another table.

Example: Students and Courses

  • One student can register for many courses
  • One course can have many students

Important: Special Handling Required

Many-to-Many relationships cannot be implemented directly in a relational database. They require a junction table (also called a bridge table or linking table) that breaks the M:M relationship into two 1:M relationships.

Junction Table Example: Enrollments

Enrollments Table (Junction)
StudentID (FK) CourseID (FK) EnrollmentDate
1001 CS101 2026-01-15
1001 CS102 2026-01-15
1002 CS101 2026-01-16

The Enrollments table contains foreign keys from both the Students and Courses tables, creating two One-to-Many relationships.

12. Importance of Relationships

Relationships are essential for building well-structured databases. They provide the following benefits:

Improve Database Structure

Relationships organize data into logical, connected tables that reflect real-world entities

Support Correct Data Retrieval

Enable accurate querying of data across multiple related tables

Enforce Data Integrity

Ensure that all references between tables are valid and consistent

Reflect Real-World Rules

Model the actual business rules and connections between entities

Best Practice

Every table should have a primary key. This is the foundation for creating relationships between tables.

13. Analyzing Relationships

When analyzing a database scenario, follow these steps to identify and design relationships correctly:

Step 1: Identify Entities

Determine the main objects in the scenario. Each object represents a possible table.

Example: Students, Courses, Departments

Step 2: Determine How Entities Are Connected

Ask: How does one entity relate to another? Identify real-world connections.

  • Students belong to departments
  • Students register for courses

Step 3: Decide the Type of Relationship

Based on the scenario, determine whether the relationship is One-to-One (1:1), One-to-Many (1:M), or Many-to-Many (M:M).

Example: One department has many students = 1:M

Step 4: Identify Primary and Foreign Keys

Identify the primary key in each table and decide which table should contain the foreign key.

  • DepartmentID = Primary Key in Departments table
  • DepartmentID = Foreign Key in Students table

Why This Skill Is Important: This analysis skill helps to design correct table relationships, avoid data redundancy, maintain data consistency, and reflect real-world rules accurately. It is the foundation of database design and will be applied in later lectures.

14. Activity 2: Why Referential Integrity?

Scenario

What problem might occur if referential integrity is not enforced between Students and Departments tables?

Discussion: Think about this for 5 minutes. Consider what happens if a student record references a department that does not exist. Work individually or discuss with your group.

Possible Problems Without Referential Integrity

  • Orphan records: A student may reference a department that has been deleted
  • Invalid data: A foreign key value may not match any primary key in the parent table
  • Data inconsistency: Reports and queries may return incorrect results
  • Loss of trust: The database becomes unreliable for decision-making

Summary and Key Takeaways

What We've Learned

  1. Relationships: A logical connection between two tables that allows data to be organized efficiently and retrieved correctly.
  2. Why Relationships Matter: They avoid data repetition, maintain consistency, connect related information, and support accurate retrieval.
  3. Related Tables: Two tables are related when they share a common field or one depends on the other for information.
  4. Primary Key in Relationships: Uniquely identifies each record and is referenced by other tables to build connections.
  5. Foreign Key in Relationships: A field in one table that refers to the primary key in another table, creating a link between them.
  6. One-to-One (1:1): Each record in one table links to only one record in another. Used rarely, often for security or data separation.
  7. One-to-Many (1:M): The most common type. One record in the parent table relates to many records in the child table.
  8. Many-to-Many (M:M): Many records relate to many records. Requires a junction table for implementation.
  9. Analyzing Relationships: Follow four steps: identify entities, determine connections, decide the type, and identify keys.
  10. Referential Integrity: Ensures that all foreign key values reference valid primary key values, maintaining data consistency.
