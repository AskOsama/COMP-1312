1. Forms

What is a Form?

A form provides a user-friendly interface to enter, view, and edit data stored in tables.

  • Forms make it easier for users to interact with data (instead of editing tables directly).
  • Forms do not store data themselves — they display and modify data that lives in tables.

Think of It This Way

A form is like a window into a table. You look through the window (form) to see and change the contents (data in the table), but the data itself is stored behind the window (in the table).

2. Reports

What is a Report?

A report presents data in a formatted, printable layout. Reports are designed for output and display — not for data entry.

Examples of What Reports Are Used For

  • Monthly sales summaries with headers and totals
  • Student grade lists organized by class
  • Inventory reports grouped by category

Forms vs. Reports — Do Not Confuse!

Forms are for data entry and editing (interactive).
Reports are for displaying and printing data (read-only output).

3. Macros

What is a Macro?

A macro automates simple, repeated tasks without programming. Macros execute predefined actions like:

  • Opening a specific form when the database starts
  • Navigating to a particular record
  • Printing a report with one click

Key Point: Macros do not require programming skills. They automate actions by selecting from predefined commands. They also do not create tables or fields — they simply perform tasks like opening forms or printing reports.

4. Bound vs. Unbound Controls

When building forms, there are two types of controls:

Bound Controls

A bound control is connected to a field in a table or query. When you edit a bound control, the change is saved directly to the underlying table.

Unbound Controls

An unbound control is not connected to any data source. It is used for decoration, titles, or labels.

Examples

  • Bound: A TextBox on a form that displays a customer's phone number from the Customer table. If you type a new number, the table is updated.
  • Unbound: A Label that displays "Customer Registration Form" as a heading. It is purely visual and has no link to any table.

Key Point:
Bound = connected to data (editing it changes the table).
Unbound = not connected to data (purely visual/informational).

A TextBox showing data from a table is bound. A Label showing a static title is unbound.

5. Filtering with AND and OR

When querying data, AND and OR combine multiple conditions:

AND Operator

All conditions must be true for a row to be included.

OR Operator

At least one condition must be true for a row to be included.

Worked Example

Consider this data:

Name Score Year
Reem 82 1
Faisal 48 1
Dana 65 2
Majed 35 1

Query: Filter where Score < 60 AND Year = 1

Name Score < 60? Year = 1? Included?
Reem No (82) Yes No
Faisal Yes (48) Yes Yes
Dana No (65) No No
Majed Yes (35) Yes Yes

Result: Faisal and Majed (both have Score < 60 and Year = 1).

Key Point: With AND, a row must satisfy every condition. If even one condition fails, the row is excluded. With OR, satisfying any one condition is enough to include the row.

6. What is Redundancy?

Redundancy

Redundancy means the same data is repeated unnecessarily across multiple records.

Example

Imagine a Product table where the SupplierName column contains "TechCorp" for 50 different products. The text "TechCorp" is stored 50 times — this is redundancy.

ProductID ProductName SupplierName
1 Laptop TechCorp
2 Monitor TechCorp
3 Keyboard TechCorp
... ... TechCorp (repeated 50 times!)

Problems Caused by Redundancy

  • Wasted storage: The same text is stored many times
  • Update anomalies: If TechCorp changes its name, you must update all 50 rows
  • Inconsistency risk: If some rows are updated but others are not, the data contradicts itself

7. The Solution — Normalization

What is Normalization?

Normalization is the process of organizing data to reduce redundancy. The solution is to:

  1. Create a separate table for the repeating data
  2. Link the tables using a Foreign Key

Normalized Design

Instead of repeating "TechCorp" 50 times:

Supplier table:

SupplierID SupplierName
10 TechCorp

Product table (with Foreign Key):

ProductID ProductName SupplierID
1 Laptop 10
2 Monitor 10
3 Keyboard 10

Now "TechCorp" is stored once in the Supplier table. If the name changes, you update it in one place.

Key Point: If you find the same text value repeated many times in a column, it probably belongs in its own table with a Foreign Key reference. This is the core idea of normalization — store each fact once, then link with keys.
